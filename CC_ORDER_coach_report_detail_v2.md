## 1. 배경 (왜 이 수정을 하는가)

코치 리포트에는 두 개의 공백이 있다. 둘 다 **데이터는 이미 손에 있는데 화면이 없을 뿐**이다.

**공백 1 - 숨은 층 미노출.** `scoring.ts`는 함정(`trap`)과 믿음(`faith`)을 계산하지만 어떤 화면에도 렌더링하지 않는다. `trap`은 프리뷰 픽스처와 "AI 입력 제외" 주석에만, `scores.faith`는 코치측 소비처가 전무하다. 설계 의도(`architecture.md §9.3`)는 함정을 소그룹 편성 기준, 믿음을 5주차 FAITH 참고로 규정하나, 인도자가 그 값을 볼 화면이 없다.

**공백 2 - 참여자 원응답 미노출.** 코치 리포트는 요약 지표(활력·나침반·준비도·간격)와 주관식만 보여준다. 항목별 원응답(각 문항에 무엇을 골랐는지)은 볼 수 없다. 그러나 인도자 실무는 원응답을 요구한다 - 예: 준비도 평균의 함정(원씽 3.0이 C6=5·C5=1인지 3·3인지), 수치와 정서의 교차 검증. 원응답이 있어야 요약 뒤의 실제 사람을 보고 디테일한 돌봄이 가능하다.

두 공백 모두 **표시(display)만 추가**해 닫는다. 채점 로직도, AI 입력 범위도 바꾸지 않는다.

근거가 되는 참여자 약속: 진단지 명문이 "여기 적는 모든 것은 세미나 인도자 한 사람만 봅니다"이다. 배정 인도자가 원응답을 보는 것은 이 약속의 이행이다(열람 범위는 §4.3에서 규정).

## 2. 절대 건드리지 말 것 (경계)

이 다섯 가지를 어기면 감사에서 반려된다.

1. **`scoring.ts` 채점 로직 불변.** `trap`·`faith`는 이미 계산돼 있다. 산식·역채점·판정 규칙을 손대지 않는다.
2. **`interpretation.ts` AI 입력 불변.** 함정·믿음은 AI에 넘기지 않는다(지어내기 방지·어휘 분리 §9.4). 원응답 전량도 AI로 배선하지 않는다(E1~E3는 기존대로 200자 클립만 유지).
3. **참여자 뷰에 노출 금지.** 참여자는 `MirrorView`/`participantMirror`를 본다. 함정·믿음·원응답 어느 것도 이 경로로 배선하지 않는다. 신규 표시는 전부 코치 전용 `ReportScreen` 계열에만 산다.
4. **의미색 남용 금지.** design_system §5 - 의미색(care 톤)은 돌봄 배너에만. 신규 패널·페이지는 중립·네이비 톤으로, 경보처럼 보이지 않게 한다.
5. **파생 점수와 원응답을 한 화면에 섞지 않는다.** '활력 11점' 같은 파생값과 원응답을 나란히 두면 역채점 문항에서 혼선이 생긴다. 원응답 페이지는 문항 원문+응답만, 해석은 앞 면에.

## 3. 확정 사실 (수정의 근거)

### 3.1 데이터 (이미 존재)

`ReportScreen`이 받는 `scores: FuturenowScores`에 다음이 이미 들어 있다(`scoring.ts`).

```ts
trap:  { D1: number; D2: number; D3: number; primary: 'D1' | 'D2' | 'D3' };  // 최고점=주함정, 동점 D1>D2>D3
faith: { F1: number | null; F2: number | null };  // 점수화 안 함, 무응답 null
```

원응답 자체도 이미 코치 리포트 페이지에 있다. `app/coach/cohort/[cohortId]/report/[responseId]/page.tsx`가 `resp.answers`(원응답 전체 객체)를 조회해 `futurenowScoring.score(resp.answers, ...)`로 넘긴다. **추가 데이터 조회 불필요** - `resp.answers`를 그대로 렌더 컴포넌트에 전달하면 된다.

### 3.2 노출 지점은 코치 전용

`ReportScreen`(`src/instruments/futurenow/report/ReportScreen.tsx`)은 코치·운영자 경로에서만 렌더된다. 참여자 경로(`app/my/cohorts/[cohortId]/report/page.tsx`)는 `MirrorView`를 쓴다. 리포트 페이지 주석도 명시한다 - "참여자는 이 임상 리포트 UI 경로 없음(§7.5 거울만)". 따라서 `ReportScreen` 계열에 추가하면 구조적으로 코치 전용이다.

### 3.3 문항 원문 소스 (원응답 페이지가 재사용)

`src/instruments/futurenow/copy.ts`가 문항 원문의 단일 소스다. 원응답 페이지는 코드가 아니라 이 원문을 렌더한다.

| export | 용도 |
|---|---|
| `itemPrompts: Record<string, string>` | 1~28번(A·NAV·B·C·D·F) 문항 원문 |
| `bipolarLabels: Record<string, {left,right}>` | 나침반 양극 레이블 |
| `askPrompts[wave].E1/E2/E3` | 주관식 문항 원문(wave별) |
| `likertLabels`·`likertCenterLabel` | 리커트 양끝·중앙 레이블 |
| `blockIntros[wave]` | 섹션 인트로(compass·now·gap·ask) |

### 3.4 라벨 부재 (`labels.ts`)

함정·믿음 라벨 상수가 없다. 기존 패턴(`GAP_AXES`·`SUBJECTIVE_LABELS`)을 따라 추가한다. 이 라벨은 강의 어휘이며 리포트 층(`labels.ts`)에 두는 것이 어휘 분리 원칙에 맞다.

## 4. 구현 지시

코치 리포트의 최종 배치를 다음 3면 구조로 만든다. **읽기 동선이 종합에서 디테일로 깊어지게** 하는 것이 핵심이다.

> **1면** 종합 해석 · 시각 리포트(기존) → **2면** 인도자 전용 패널(주 함정·믿음) → **3면** 참여자 원응답(섹션별 문항+응답)

- 화면에서 '면'은 순차 섹션(2·3면은 접이식 권장)으로, PDF에서는 개행 페이지로 구현한다.
- 순서를 뒤집지 않는다. 원응답(3면)이 종합(1면)·인도자 패널(2면) 앞에 오면 안 된다.

### Phase 1 (필수) - 2면: 인도자 전용 패널 (함정·믿음)

#### 4.1a `labels.ts`에 라벨 상수 추가

```ts
// 함정 유형(D1~D3) - 인도자 전용 강의 어휘. 참여자 미노출.
export const TRAP_AXES = [
  { code: 'D1', label: '관성' },
  { code: 'D2', label: '준비' },
  { code: 'D3', label: '안주' },
] as const;

// 믿음의 자리(F1·F2) - 점수화 안 함, 목회적 참고.
export const FAITH_LABELS = { F1: '의미', F2: '실행' } as const;
```

#### 4.1b 신규 패널 컴포넌트

`ReportScreen` 하단(주관식 다음)에 배치. 다른 패널과 시각적으로 구분(구분선 + "인도자 전용" 캡션 헤더).

| 구획 | 내용 | 표시 |
|---|---|---|
| 헤더 | "인도자 전용 · 참여자에게 보이지 않습니다" | 캡션, 중립색 |
| 함정 유형 | 주 함정(primary → TRAP_AXES 라벨) 강조 + D1·D2·D3 원점수 | 주 함정 굵게, 나머지 회색. "소그룹 편성 참고" 캡션 |
| 믿음의 자리 | F1 의미 · F2 실행 → 값 또는 "무응답" | "점수화하지 않는 목회적 신호" 캡션. null이면 "무응답" |
| 주의 | "참여자에게 '함정' 같은 낱말을 그대로 쓰지 않습니다." | 캡션 |

톤: `--color-surface-2`·`--color-border`·`--color-text-secondary` 중립 토큰만. care 톤 금지.

### Phase 2 (필수) - 3면: 참여자 원응답 페이지

`resp.answers`를 섹션별 문항 원문 + 응답으로 렌더하는 신규 컴포넌트를 만든다. 코치 리포트 페이지가 `resp.answers`(와 wave)를 이 컴포넌트에 전달한다.

**다섯 조건(전부 충족해야 함):**

1. **순서 - 해석 뒤에 둔다.** 1면·2면 다음 3면. 앞에 두지 않는다.
2. **구성 - 코드 덤프가 아니라 섹션별 문항 원문 + 응답.** `copy.ts`의 `itemPrompts`·`bipolarLabels`·`askPrompts`를 재사용한다. 예: "'열심히 살고는 있지만, 제자리걸음이라는 느낌이 든다.' → 매우 그렇다". 리커트는 `likertLabels`로 응답값을 문구화(1='전혀 아니다' … 5='매우 그렇다'), 나침반은 좌·우 레이블과 함께 표시, 간격은 "N / 10", 주관식은 전문(全文) 그대로.
3. **섹션 구획.** 진단지 구조를 따른다 - 나침반(NAV1~4) · 지금의 나(A·C·D 리커트) · 믿음(F1·F2, 무응답 표기) · 다섯 영역의 간격(B1~B5) · 주관식(E1~E3). `blockIntros`를 섹션 헤더 참고로 쓸 수 있다.
4. **가드레일 - 함정·믿음 패널과 동일.** 참여자 미노출·AI 미입력. 코치 전용 경로에만. 원응답에 D코드·F코드 문항이 날것으로 포함되지만, 이는 코치 전용 경로라 §2 경계 안이다.
5. **파생 점수 비혼합.** 이 페이지에는 활력 지수·GROW 축값 등 파생 점수를 넣지 않는다(§2-5). 원문+응답만.

**표시 유의:** 역채점 문항이라도 별도 표식 없이 문항 원문 그대로 보여준다. 문장 자체가 자기설명적이므로(문장을 읽으면 방향이 이해됨) 인도자가 오해하지 않는다. '역채점'·점수 변환을 이 페이지에 노출하지 않는다.

### Phase 3 (후속·선택) - 로스터 주 함정 태그

소그룹 편성을 한눈에 돕기 위해 로스터 멤버 행에 주 함정 태그(관성/준비/안주)를 붙인다. `rosterModel.ts`가 멤버별 응답을 채점해 `trap.primary`를 노출하도록 확장한다. 멤버 수만큼 채점하므로 성능을 검토한다. Phase 1·2 완료·검증 후 별도 지시로 착수해도 된다.

**Phase 1·2·3을 분리 커밋한다.**

## 4.3 열람 범위 - 접근 제어 [최박사 확인 항목]

원응답(3면)은 가장 민감한 형태의 데이터다. 진단지 약속은 "인도자 한 사람만 봅니다"이다. 이를 기준으로 **기본값을 다음으로 정한다.**

- **원응답(3면)**: 해당 참여자에게 배정된 인도자(코치) role에 한해 열람. 운영자(수퍼바이저)는 열람 제한.
- **종합 해석(1면)·인도자 패널(2면)**: 기존 리포트 경로 권한(차수 코치+운영자+본인) 유지.

> **[최박사 확인]** 운영자 수퍼바이저에게도 원응답 열람을 허용하려면, 이 조건을 리포트 경로 권한과 동일하게 완화한다(구현이 더 단순해짐). 어느 쪽이든 **최근 도입된 개인정보 동의(커밋 `440f985`) 문구와 정합**해야 한다 - 운영자 열람을 허용한다면 동의 문구가 그 범위를 포함하는지 확인이 선행된다.

클코1은 기본값(배정 코치만 원응답)으로 구현하되, 위 확인 결과가 "운영자 포함"이면 그에 맞춰 게이트를 완화한다.

## 5. 인수 기준 (테스트)

1. **코치 리포트에 함정·믿음이 뜬다.** `ReportScreen.test`에 주 함정 라벨·F1/F2(또는 "무응답") 렌더 단언 추가.
2. **코치 리포트에 원응답이 문항 원문으로 뜬다.** 원응답 컴포넌트 테스트 - 코드가 아니라 `itemPrompts`/`bipolarLabels`/`askPrompts` 원문과 응답이 렌더됨을 단언.
3. **배치 순서.** 원응답이 종합·인도자 패널 다음에 온다는 것을 렌더 순서로 단언.
4. **참여자 미러에 함정·믿음·원응답이 없다.** `participantMirror` 산출·`MirrorView` 렌더에 D1~D3·F1·F2·"관성/준비/안주"·원응답 문항 목록이 없음을 단언(회귀 방지).
5. **AI 입력 불변.** `interpretation.ts`의 `buildInterpretationInput`에 함정·믿음·원응답 전량이 포함되지 않음(E1~E3는 기존 200자 클립만)을 단언.
6. **null·빈 응답 처리.** 믿음 무응답(null)은 "무응답"으로, 주관식 빈칸은 빈칸대로 표기하며 런타임 에러 없음.
7. **열람 범위.** §4.3 기본값 기준, 배정 코치는 원응답이 보이고, (기본값에서) 운영자 경로에서는 원응답이 렌더되지 않음을 단언. [최박사 확인이 "운영자 포함"이면 이 테스트를 그에 맞게 조정.]
8. **채점 불변.** `scoring.test` 무수정 통과.

## 7. 이 수정과 인도자 가이드의 관계 (참고)

완료되면 가이드의 다음이 사실이 되어, 지휘부가 해당 서술을 실제 화면명으로 정정한다.
