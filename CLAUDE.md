# CLAUDE.md — 퓨처나우 플랫폼 작업 규율

> 이 파일은 클로드코드의 작업 규율이다. 매 작업 전 `architecture.md`와 함께 읽는다.
> `architecture.md` = 확정 사양(단일 진실) · `plan.md` = 보류·향후 · 본 파일 = 작업 방식.

---

## 0. 문서 우선순위

1. `architecture.md`를 먼저 정독한다. 그것이 단일 진실이다.
2. 본 파일의 규율을 지킨다.
3. 충돌·의문이 생기면 **임의로 결정하지 말고 지휘부에 질문**한다. 특히 `/contracts` 관련.

---

## 1. 계약 우선 (Contract-first)

- `/src/contracts`의 타입은 시스템의 척추다. **변경은 지휘부 승인 후에만** 한다.
- `/src/core`는 `/src/instruments`를 **참조하지 않는다.** 진단도 코어를 직접 참조하지 않는다. 양쪽 다 `/src/contracts`만 바라본다.
- 계약을 우회하는 지름길(코어에서 진단 내부를 들여다보기, 진단끼리 코드 공유)을 만들지 않는다. 이 경계가 흐려지면 플랫폼 전체가 무너진다.

## 2. 코어는 채점하지 않는다

- 코어(`/src/core`)에 채점·리포트·진단 고유 로직을 **한 줄도** 넣지 않는다. 코어는 인증·차수·코치·응답봉투·알림·공용UI만 운영한다.
- 채점·리포트·알림 규칙은 전부 `/src/instruments/futurenow` 안에 둔다.

## 3. 소스코드 보존 규율 (지휘부 원칙)

- 기존 코드를 수정할 때는 **개선에 필요한 부분만** 고친다. 그 외 부분을 임의로 축약·요약·생략·재작성하지 않는다.
- 리팩터링이 필요하면 범위를 먼저 보고하고 승인받는다.

## 4. SAIL을 건드리지 않는다

- 현재 운영 중인 SAIL은 별도 작업공간(클코3) 소관이다. 본 작업공간에서 SAIL 코드를 수정하지 않는다.
- SAIL의 추상 이관은 코어 패키지 추출 이후다(`plan.md` §1). 지금 SAIL을 추상에 맞춰 미리 고치지 않는다.
- SAIL은 **참고·차용의 원본**으로만 본다(스키마·RLS 헬퍼·단계형 흐름·나침반 차트).

## 5. 마이그레이션 규율

- 마이그레이션은 `/supabase/migrations`에 타임스탬프 파일로 추가한다.
- **이미 적용된 마이그레이션 파일을 수정하지 않는다.** 변경은 항상 새 마이그레이션으로.
- RLS는 `SECURITY DEFINER` 헬퍼(`is_admin` 등) 패턴을 따른다(architecture §6.1). 정책끼리 테이블을 직접 참조해 재귀를 만들지 않는다.
- SAIL과 **동일 Supabase 프로젝트**를 공유한다. 기존 SAIL 테이블(`users` 등)과 충돌하지 않도록, 공유 테이블 변경은 반드시 지휘부 확인 후.

## 6. 스택 버전 주의

- Next.js 16은 학습 데이터와 다른 breaking change가 많다. 코드를 짜기 전에 `node_modules/next/dist/docs/`의 해당 가이드를 읽는다(SAIL `AGENTS.md` 교훈).
- React 19 · Tailwind 4 · Supabase SSR(`@supabase/ssr`) 최신 규약을 따른다.

## 7. 측정·강의 어휘 분리

- 문항(`Item.prompt`)에 구인명·STEP·강의 어휘('시들음'·'원씽' 등)를 **노출하지 않는다.**
- `code`·`polarity`·구인은 참여자 렌더 경로에 두지 않는다. 그 명명은 리포트(B③)에서만.

## 8. 디자인 시스템 — 확정 전 UI 임의 디자인 금지

- `design_system.md`가 도착하기 전까지 **UI를 임의로 디자인하지 않는다.** 색·폰트·레이아웃·컴포넌트 시안은 지휘부가 확정해 전달한다.
- 그 전 단계에서 UI가 불가피하면, **기능 골격만(스타일 미적용 placeholder)** 만들고 디자인은 비워 둔다.

## 9. 검증

- 채점(B②)은 **단위테스트 필수.** 전용노선과 산출이 동일함을 테스트로 증명한다(architecture §9.3 규칙별 케이스).
- 경계에서 `answers`·`subjectProfile`를 zod로 검증한다. DB는 느슨(JSONB), 코드 경계는 엄격.
- RLS 격리(실명제·전화번호 운영자 전용)를 권한별로 테스트한다.

## 10. 보안

- 전화번호는 `user_contacts`에 격리하고 `getPhone` 게이트로만 접근한다. `CoreUser`에 전화번호를 싣지 않는다.
- `responses`·`alerts`는 불변(UPDATE/DELETE 정책 없음).
- 비밀키·환경변수를 코드·로그에 노출하지 않는다.

## 11. 작업 보고 (보고 규격 — 명문화 2026-06-26)

모든 작업 완료 보고는 다음을 **빠짐없이** 포함한다.

1. **구현/변경 파일 목록과 각 항목 상태**(완·부분·보류).
2. **검증 결과**: 타입(`tsc`)·린트(`eslint`)·테스트 통과/실패 수. 보안 관련은 **역할별 실측 여부**(RLS 격리가 역할별로 실제로 검증됐는지).
3. **`architecture.md` 반영 항목**(있으면) — ADR·§4·§5·§7·plan 확정 포함.
4. **계약(`/contracts`) 대비 이탈 여부와 사유** — 임의 변경 금지, 필요시 질문(변경은 견고화 방향 + 지휘부 승인 후에만).
5. **막힌 지점·판단 필요 사항**.

- **작업 완료 보고는 GitHub 푸시를 전제로 하며, 커밋 해시·변경 파일을 포함한다**(지휘부가 매 단계 클론·감리). 비밀(`.env.local`·service_role 등) 추적 제외를 push 전 재확인한다.
- 컨텍스트 한계가 가까워지면 핸드오버 요약을 남긴다.
